1️⃣ GET vs POST: Choosing the Right Method

In the previous session, we learned about HTTP methods. Now let's dive deep into the two most common methods for working with data: GET and POST.

When to Use GET

GET requests are used to retrieve data from the server. They:

  • Send data in the URL (query parameters)
  • Are cached by browsers
  • Have URL length limitations (~2KB)
  • Are idempotent (safe to call multiple times)
  • Should not modify server state

✓ Good Use Cases for GET

# Retrieve user by ID GET /api/users/123 # Search with filters GET /api/products?category=electronics&price_max=500 # Get paginated results GET /api/posts?page=1&limit=10 # Get user profile GET /api/me/profile

When to Use POST

POST requests are used to create new data on the server. They:

  • Send data in the request body
  • Can send large amounts of data (limited by server config)
  • Are not cached by default
  • Typically return HTTP 201 (Created)
  • Change server state

✓ Good Use Cases for POST

# Create new user POST /api/users { "name": "John", "email": "john@example.com", "password": "secure_password" } # Submit form data POST /api/contact { "subject": "Help needed", "message": "I have a question", "email": "user@example.com" } # Upload file with metadata POST /api/documents { "title": "My Document", "file": "...", "tags": ["important", "review"] }
Aspect GET POST
Purpose Retrieve data Create/Submit data
Data Location URL (query params) Request body
Data Size Limited (~2KB) Large (configurable)
Idempotent Yes (safe) No (creates resources)
Cached Yes (by default) No (by default)
Response Status 200 OK 201 Created

2️⃣ Understanding Request Bodies

A request body is the data sent with a request. In FastAPI, we use Pydantic models to define and validate request bodies.

What is a Request Body?

The request body is the "payload" of data sent to the server. It's typically formatted as JSON and contains all the data needed to create or update a resource.

Key Concept: A request body is optional for GET requests but required for POST, PUT, and PATCH requests. The body contains structured data that the server needs to process.

Anatomy of a Request with Body

Complete HTTP Request with Body

POST /api/users HTTP/1.1 Host: api.example.com Content-Type: application/json Content-Length: 89 { "name": "Alice Johnson", "email": "alice@example.com", "age": 28, "is_active": true }

Breaking this down:

  • Request Line: POST /api/users HTTP/1.1 - Method, path, protocol
  • Headers: Metadata about the request (content type, length, etc.)
  • Body: The actual JSON data to be sent

Defining a Pydantic Model for Request Body

Basic Pydantic Model

from pydantic import BaseModel class User(BaseModel): name: str email: str age: int is_active: bool = True # Optional with default # FastAPI automatically validates request bodies @app.post("/api/users") def create_user(user: User): # user.name, user.email, user.age are guaranteed valid return {"message": "User created", "user": user}
Key Concept: When you define a parameter with a Pydantic model type, FastAPI automatically:
  • Expects JSON in the request body
  • Validates each field against the type annotation
  • Converts JSON strings to Python objects
  • Returns 422 error if validation fails

3️⃣ Pydantic Models: Advanced Validation

Pydantic is a Python library for data validation using type annotations. It ensures that data is correct before your code processes it.

Type Validation

The simplest form of validation is using Python type hints:

Type Validation Example

from pydantic import BaseModel class Product(BaseModel): name: str # Must be a string price: float # Must be a number stock: int # Must be an integer in_stock: bool # Must be true or false # Valid request: # {"name": "Laptop", "price": 999.99, "stock": 5, "in_stock": true} # Invalid request (will return 422 error): # {"name": "Laptop", "price": "expensive", "stock": 5, "in_stock": true} # ↑ "expensive" is not a number

Optional Fields

Use Optional to make fields optional:

Optional Fields

from typing import Optional from pydantic import BaseModel class BlogPost(BaseModel): title: str content: str author: str tags: Optional[list] = None # Optional, defaults to None views: Optional[int] = 0 # Optional, defaults to 0 # All valid: # {"title": "Hello", "content": "...", "author": "John"} # {"title": "Hello", "content": "...", "author": "John", "tags": ["python"]} # {"title": "Hello", "content": "...", "author": "John", "tags": [], "views": 100}

Field Validation with Constraints

Use the Field class to add validation constraints:

Using Field for Validation

from pydantic import BaseModel, Field class User(BaseModel): name: str = Field(..., min_length=1, max_length=100) email: str = Field(...) age: int = Field(..., ge=0, le=150) # >= 0, <= 150 username: str = Field(..., min_length=3, max_length=20) bio: str = Field(default="", max_length=500) # Valid: # {"name": "John", "email": "john@example.com", "age": 30, "username": "john_doe"} # Invalid (age out of range): # {"name": "John", "email": "john@example.com", "age": 200, "username": "john_doe"}

Common Field Constraints

Constraint Purpose Example
min_length Minimum string length min_length=3
max_length Maximum string length max_length=50
gt Greater than gt=0
ge Greater than or equal ge=0
lt Less than lt=100
le Less than or equal le=100
regex Regular expression pattern regex="^[a-z]+$"

Nested Models

You can nest Pydantic models to represent complex data structures:

Nested Models Example

from pydantic import BaseModel class Address(BaseModel): street: str city: str country: str zip_code: str class Customer(BaseModel): name: str email: str address: Address # Nested model # Request body: # { # "name": "John", # "email": "john@example.com", # "address": { # "street": "123 Main St", # "city": "New York", # "country": "USA", # "zip_code": "10001" # } # } @app.post("/customers") def create_customer(customer: Customer): return customer # All data is validated

4️⃣ Error Handling and Status Codes

Proper error handling is crucial for API reliability. Users need to know what went wrong and how to fix it.

HTTP Status Codes for Data Submission

Success Status Codes

Code Name Use Case
200 OK Request succeeded, data returned
201 Created Resource successfully created
202 Accepted Request accepted for processing
204 No Content Success, no data to return

Client Error Status Codes

Code Name Use Case
400 Bad Request Invalid request format or data
401 Unauthorized Authentication required or failed
403 Forbidden User not authorized for this action
404 Not Found Resource doesn't exist
409 Conflict Data conflict (e.g., duplicate email)
422 Unprocessable Entity Validation error

Validation Errors (422)

When Pydantic validation fails, FastAPI automatically returns a 422 error with details:

422 Validation Error Response

# Request with invalid data: POST /api/users { "name": "John", "email": "not-an-email", "age": "thirty" } # Response (422 Unprocessable Entity): { "detail": [ { "loc": ["body", "age"], "msg": "value is not a valid integer", "type": "type_error.integer" } ] }

Custom Error Responses

You can create custom error handling for specific scenarios:

Custom Error Handling

from fastapi import HTTPException @app.post("/users") def create_user(user: User): # Check for duplicate email if user_exists(user.email): raise HTTPException( status_code=409, detail="Email already registered" ) # Check age requirement if user.age < 18: raise HTTPException( status_code=400, detail="User must be at least 18 years old" ) return {"message": "User created"}

Processing Request Data

Once validation passes, process the data safely:

Safely Processing Request Data

@app.post("/articles") def create_article(article: Article): # Data is already validated at this point try: # Save to database db_article = save_article_to_database(article) # Return success response return { "status": "created", "id": db_article.id, "message": "Article published successfully" } except DatabaseError as e: raise HTTPException( status_code=500, detail="Failed to save article" )
⚠️ Security Note: Always validate user input, even with Pydantic. Never trust client data. Use Pydantic for type validation, but also implement business logic validation (like checking for duplicates, age requirements, etc.).

5️⃣ Building a Complete API Endpoint

Let's build a real-world example: an API for managing a todo list that combines everything we've learned.

Application Design

Todo Application Structure

from fastapi import FastAPI, HTTPException from pydantic import BaseModel, Field from typing import Optional, List from datetime import datetime app = FastAPI( title="Todo API", description="Manage your todos with this API", version="1.0.0" ) # Pydantic models class TodoBase(BaseModel): title: str = Field(..., min_length=1, max_length=200) description: Optional[str] = Field(None, max_length=1000) priority: int = Field(default=1, ge=1, le=5) # 1-5 scale is_completed: bool = False class TodoCreate(TodoBase): pass # Used for POST requests class Todo(TodoBase): id: int created_at: datetime # Mock database todos_db = {} next_id = 1 # Endpoints @app.post("/todos", status_code=201) def create_todo(todo: TodoCreate): global next_id # Validate: no empty title if not todo.title.strip(): raise HTTPException( status_code=400, detail="Title cannot be empty or whitespace only" ) # Create todo new_todo = Todo( id=next_id, title=todo.title, description=todo.description, priority=todo.priority, is_completed=todo.is_completed, created_at=datetime.now() ) todos_db[next_id] = new_todo next_id += 1 return { "status": "created", "todo": new_todo } @app.get("/todos") def get_todos(): return list(todos_db.values()) @app.get("/todos/{todo_id}") def get_todo(todo_id: int): if todo_id not in todos_db: raise HTTPException( status_code=404, detail=f"Todo with id {todo_id} not found" ) return todos_db[todo_id] @app.put("/todos/{todo_id}") def update_todo(todo_id: int, todo: TodoCreate): if todo_id not in todos_db: raise HTTPException( status_code=404, detail=f"Todo with id {todo_id} not found" ) todos_db[todo_id].title = todo.title todos_db[todo_id].description = todo.description todos_db[todo_id].priority = todo.priority todos_db[todo_id].is_completed = todo.is_completed return { "status": "updated", "todo": todos_db[todo_id] } @app.delete("/todos/{todo_id}", status_code=204) def delete_todo(todo_id: int): if todo_id not in todos_db: raise HTTPException( status_code=404, detail=f"Todo with id {todo_id} not found" ) del todos_db[todo_id] return None

Testing the API

You can test this API using curl or any HTTP client:

Example API Calls

# Create a todo curl -X POST http://localhost:8000/todos \ -H "Content-Type: application/json" \ -d '{"title": "Learn FastAPI", "priority": 5}' # Get all todos curl http://localhost:8000/todos # Get specific todo curl http://localhost:8000/todos/1 # Update todo curl -X PUT http://localhost:8000/todos/1 \ -H "Content-Type: application/json" \ -d '{"title": "Learn FastAPI", "priority": 4, "is_completed": true}' # Delete todo curl -X DELETE http://localhost:8000/todos/1

6️⃣ Real-World Patterns and Best Practices

Pattern 1: Bulk Operations

Accept multiple items in a single request:

Bulk Create Example

@app.post("/bulk/items") def create_items_bulk(items: List[ItemCreate]): # Validate each item created_items = [] for item in items: new_item = save_item(item) created_items.append(new_item) return { "created_count": len(created_items), "items": created_items } # Request: # [ # {"name": "Item 1", "price": 10.0}, # {"name": "Item 2", "price": 20.0} # ]

Pattern 2: Partial Updates (PATCH)

Allow updating only specific fields:

PATCH for Partial Updates

from typing import Optional class UserUpdate(BaseModel): name: Optional[str] = None email: Optional[str] = None age: Optional[int] = None @app.patch("/users/{user_id}") def partial_update_user(user_id: int, user_update: UserUpdate): existing_user = get_user(user_id) # Only update provided fields if user_update.name: existing_user.name = user_update.name if user_update.email: existing_user.email = user_update.email if user_update.age: existing_user.age = user_update.age save_user(existing_user) return existing_user # Request (only update name and email): # {"name": "New Name", "email": "newemail@example.com"}

Pattern 3: Response Models

Return only specific fields in responses:

Response Models for Security

class UserResponse(BaseModel): id: int name: str email: str # Note: password is NOT included class UserCreate(BaseModel): name: str email: str password: str @app.post("/users", response_model=UserResponse) def create_user(user: UserCreate): # Even if UserCreate has password, # response will only include id, name, email new_user = save_user(user) return new_user

Best Practices

✓ Best Practices for Sending Data:
  • Use appropriate HTTP status codes (201 for created, 200 for success)
  • Validate all input data with Pydantic models
  • Return meaningful error messages with proper status codes
  • Never expose sensitive data (passwords, keys) in responses
  • Use PATCH for partial updates, PUT for full replacements
  • Handle edge cases (duplicates, invalid references, etc.)
  • Log important operations for debugging
  • Test your endpoints with various input scenarios

7️⃣ Summary: Key Takeaways

What We Learned

📌 GET vs POST

GET retrieves data via URL parameters, POST sends data in the request body to create resources. Choose based on your use case and data size.

📌 Request Bodies

Request bodies contain data sent to the server, typically as JSON. They're defined using Pydantic models that automatically validate and convert the data.

📌 Pydantic Validation

Pydantic validates data types, field constraints, and complex nested structures. Use Field() for advanced validation like min/max lengths and numeric ranges.

📌 Error Handling

Return appropriate HTTP status codes and error messages. FastAPI automatically handles validation errors (422), but you should handle business logic errors (409 conflict, 400 bad request).

📌 Complete Endpoints

Build endpoints by combining Pydantic models, validation logic, and proper status codes. Test thoroughly with various input scenarios.

📌 Real-World Patterns

Use bulk operations for multiple items, PATCH for partial updates, and response models to control what data is returned.
