🔧 HTTP Methods: The Verbs of the Web

HTTP methods define the action you want to perform on a resource. Think of them as verbs in a sentence. The most common REST APIs use these 4-7 methods.

1. GET - Retrieve Data

🔍 Purpose: Read and retrieve data without modifying it

Key Characteristics:

  • Safe - doesn't change server state
  • Idempotent - calling multiple times returns the same result
  • No request body (data in URL parameters)
  • Cacheable - results can be cached by browsers/proxies
  • Should not have side effects

Examples:

GET /api/users
GET /api/users/123
GET /api/posts?category=technology&limit=10
GET /api/weather?lat=40.71&lon=-74.01

2. POST - Create Data

➕ Purpose: Create a new resource on the server

Key Characteristics:

  • Not safe - creates new data on server
  • Not idempotent - calling multiple times creates duplicates
  • Has a request body with the data to create
  • Returns 201 Created status code
  • Location header points to new resource

Examples:

POST /api/users
{
  "name": "Jane Doe",
  "email": "jane@example.com",
  "password": "secure123"
}

POST /api/blog/posts
{
  "title": "Getting Started with APIs",
  "content": "...",
  "author_id": 42
}

3. PUT - Replace Entire Resource

♻️ Purpose: Completely replace a resource with new data

Key Characteristics:

  • Not safe - modifies data on server
  • Idempotent - calling multiple times has same effect
  • Requires specifying the complete resource
  • Replaces the entire resource (not partial updates)
  • Returns 200 OK or 204 No Content

Examples:

PUT /api/users/123
{
  "name": "Jane Smith",
  "email": "jane.smith@example.com",
  "role": "admin",
  "status": "active"
}
Note: PUT replaces the ENTIRE resource. If you omit a field, it may be set to null or default value.

4. DELETE - Remove Data

🗑️ Purpose: Delete a resource from the server

Key Characteristics:

  • Not safe - removes data from server
  • Idempotent - deleting twice has same result
  • No request body needed
  • Returns 204 No Content or 200 OK
  • Sometimes returns deleted resource in response

Examples:

DELETE /api/users/123

DELETE /api/posts/456

DELETE /api/comments/789

5. PATCH - Partial Update

🔧 Purpose: Update specific fields of a resource (partial update)

Key Characteristics:

  • Not safe - modifies data on server
  • May or may not be idempotent
  • Only include fields you want to change
  • More efficient than PUT for small updates
  • Returns 200 OK with updated resource

Examples:

PATCH /api/users/123
{
  "email": "newemail@example.com"
}

PATCH /api/products/456
{
  "price": 29.99
}
PUT vs PATCH: Use PUT to replace the entire resource, PATCH to update only specific fields.

6. HEAD - Like GET but No Response Body

📋 Purpose: Get headers only, useful for checking resource existence or size

Key Characteristics:

  • Safe - doesn't change data
  • Idempotent
  • Returns same headers as GET but no body
  • Useful for checking if resource exists
  • Can check Last-Modified and Content-Length headers

Example Use Cases:

  • Check if file exists before downloading
  • Verify resource modification time
  • Check content length without downloading

7. OPTIONS - Describe Communication Options

ℹ️ Purpose: Get information about what methods are allowed for a resource

Key Characteristics:

  • Safe - doesn't change data
  • Used by browsers for CORS pre-flight checks
  • Returns Allow header with permitted methods
  • Useful for API discovery

Example Response:

OPTIONS /api/users/123

Response Headers:
Allow: GET, POST, PUT, DELETE, PATCH, OPTIONS

HTTP Methods Comparison Table

Method Safe? Idempotent? Cacheable? Has Body? Purpose
GET ✓ Yes ✓ Yes ✓ Yes ✗ No Retrieve data
POST ✗ No ✗ No ✗ No* ✓ Yes Create data
PUT ✗ No ✓ Yes ✗ No* ✓ Yes Replace entire resource
DELETE ✗ No ✓ Yes ✗ No* ✗ No Delete resource
PATCH ✗ No Maybe ✗ No* ✓ Yes Partial update
HEAD ✓ Yes ✓ Yes ✓ Yes ✗ No Get headers only
OPTIONS ✓ Yes ✓ Yes ✓ Yes ✗ No Describe options

📊 Status Codes: API Response Signals

HTTP status codes are 3-digit numbers that tell the client what happened with their request. They're grouped into 5 categories.

1xx - Informational (100-199)

The request was received and the server is processing it. Rarely used in modern APIs.

  • 100 Continue - Client should continue sending request body
  • 101 Switching Protocols - Changing to WebSocket, HTTP/2, etc.

2xx - Success (200-299)

✅ The request was successful. The client can trust the response.

Code Name Meaning Use Case
200 OK Request successful, response has body GET request returns data, POST request succeeds
201 Created Resource successfully created POST request creates new user/post/item
202 Accepted Request accepted but processing continues Async job started, background processing
204 No Content Success but no body in response DELETE request, PATCH with no response needed
206 Partial Content Only part of resource is returned Resumable downloads, Range requests

3xx - Redirection (300-399)

⛔ The resource has moved or the client needs to take additional action.

Code Name Meaning Use Case
301 Moved Permanently Resource permanently moved to new URL API endpoint restructured (old endpoint removed)
302 Found Resource temporarily at different URL Temporary redirect, load balancing
304 Not Modified Resource not changed since last request Caching: client has latest version
307 Temporary Redirect Temporary redirect, maintain HTTP method Safe redirection for POST/PUT requests

4xx - Client Error (400-499)

❌ The client made a bad request or doesn't have permission.

Code Name Meaning Use Case
400 Bad Request Server can't understand request (invalid data) Malformed JSON, missing required fields
401 Unauthorized Authentication required but missing/invalid API key missing, token expired, invalid credentials
403 Forbidden Authenticated but not authorized to access User doesn't have permission (insufficient role)
404 Not Found Resource doesn't exist User ID doesn't exist, endpoint URL wrong
409 Conflict Request conflicts with current state Duplicate email during signup, version conflict
429 Too Many Requests Rate limit exceeded Exceeded API quota, too many requests in timeframe

5xx - Server Error (500-599)

🔴 The server encountered an error while processing the valid request.

Code Name Meaning Use Case
500 Internal Server Error Generic server error (code crashed) Unhandled exception, database query error
501 Not Implemented Server doesn't support this method Endpoint doesn't implement requested HTTP method
502 Bad Gateway Server got invalid response from upstream Proxy/load balancer issue, backend offline
503 Service Unavailable Server temporarily down for maintenance Scheduled maintenance, server overloaded
504 Gateway Timeout Server didn't get response in time Slow database query, backend timeout
Remember: 2xx = Success ✓ | 3xx = Redirect ⛔ | 4xx = Client Error ❌ | 5xx = Server Error 🔴

📝 Headers: The Metadata of Communication

Headers are key-value pairs that provide metadata about the request or response. They tell the server/client additional information about how to handle the data.

Request Headers (Client → Server)

🔐 Authentication Headers
  • Authorization - Sends credentials (API key, JWT token, etc.)
    Authorization: Bearer eyJhbGc...
    Authorization: Basic dXNlcjpwYXNz
    Authorization: ApiKey sk-1234567890
  • X-API-Key - Alternative API key header
    X-API-Key: sk_live_1234567890abcdef
📋 Content Headers
  • Content-Type - Tells server what format data is in
    Content-Type: application/json
    Content-Type: application/x-www-form-urlencoded
    Content-Type: multipart/form-data
    Content-Type: text/plain
  • Content-Length - Size of request body in bytes
    Content-Length: 348
  • Accept - What format client wants in response
    Accept: application/json
    Accept: application/xml
    Accept: text/html
🖥️ Request Context Headers
  • User-Agent - Identifies the client (browser, app, etc.)
    User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
    User-Agent: MyApp/1.0 (iOS 15.0)
  • Host - Domain name of the server
    Host: api.example.com
  • Origin - Where the request came from (for CORS)
    Origin: https://myapp.com
  • Referer - Previous page URL
    Referer: https://google.com/search
⚡ Caching Headers
  • Cache-Control - How long to cache the response
    Cache-Control: max-age=3600
    Cache-Control: no-cache
    Cache-Control: private, max-age=300
  • If-Modified-Since - Only get if changed since date
    If-Modified-Since: Wed, 21 Oct 2024 07:28:00 GMT
  • ETag - Unique identifier for resource version (sent in response)
    ETag: "123456789"
🔗 Custom Headers

APIs often define their own headers (prefixed with X-):

  • X-Request-ID - Unique identifier for tracking
    X-Request-ID: abc-123-def-456
  • X-Rate-Limit-Remaining - Requests left before rate limit
    X-Rate-Limit-Remaining: 45
  • X-Custom-Header - Any custom data
    X-Client-Version: 2.1.0

Response Headers (Server → Client)

📊 Response Info Headers
  • Content-Type - Format of response data
    Content-Type: application/json; charset=utf-8
  • Content-Length - Size of response body
    Content-Length: 1234
  • Server - Server software
    Server: nginx/1.24.0
🔄 Caching Headers
  • Cache-Control - How client should cache
    Cache-Control: public, max-age=3600
    Cache-Control: private, no-cache
  • Last-Modified - When resource was last changed
    Last-Modified: Wed, 21 Oct 2024 07:28:00 GMT
  • ETag - Unique version of resource
    ETag: "33a64df5425b"
🔐 Security Headers
  • Access-Control-Allow-Origin - CORS: who can access
    Access-Control-Allow-Origin: https://myapp.com
    Access-Control-Allow-Origin: *
  • Access-Control-Allow-Methods - CORS: allowed HTTP methods
    Access-Control-Allow-Methods: GET, POST, PUT, DELETE
  • Set-Cookie - Store cookie on client
    Set-Cookie: session_id=abc123; Path=/; HttpOnly
  • Strict-Transport-Security - Enforce HTTPS
    Strict-Transport-Security: max-age=31536000; includeSubDomains
📍 Redirect Headers
  • Location - Where to redirect (used with 3xx codes)
    Location: https://api.example.com/v2/users

🔑 API Keys: Security & Access Control

API keys are credentials that authenticate your application and control access to API services. They're essential for security and rate limiting.

What is an API Key?

Definition: An API key is a unique string of characters that identifies your application to the API service. It proves you have permission to use the API and helps the server track your usage.

Types of API Keys

1️⃣ Public Keys

Can be exposed in frontend code (not sensitive).

  • Frontend/browser analytics keys
  • Stripe publishable keys
  • Google Maps API keys (with domain restrictions)
const publicKey = "pk_live_abc123def456";
// Safe to put in frontend code (with restrictions)

2️⃣ Secret Keys

Must NEVER be exposed (highly sensitive).

  • Backend API keys
  • Database credentials
  • Stripe secret keys
  • OpenAI API keys
const secretKey = "sk_live_xyz789abc123";
// MUST be kept secret - store in environment variables!
// Use: process.env.OPENAI_API_KEY

3️⃣ Bearer Tokens / JWT

Temporary tokens for user sessions.

  • OAuth tokens
  • JWT (JSON Web Tokens)
  • Session tokens
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
// Token expires after certain time

How API Keys Work in Requests

Method 1: Authorization Header

GET /api/users
Authorization: Bearer sk_live_1234567890abcdef

// Server validates the key and processes request

Method 2: X-API-Key Header

GET /api/data
X-API-Key: sk_live_1234567890abcdef

// Common in third-party APIs

Method 3: Query Parameter

GET /api/weather?lat=40.71&lon=-74.01&key=sk_live_1234567890abcdef

// Less secure (exposes key in URL)

API Key Best Practices

🔒 Security Rules:
  • Never commit API keys to GitHub (use .env files)
  • Rotate keys periodically (create new ones, delete old)
  • Use different keys for development, staging, production
  • Store keys in environment variables (process.env.API_KEY)
  • Use secrets management tools (AWS Secrets Manager, HashiCorp Vault)
  • Monitor key usage for suspicious activity
  • Immediately revoke compromised keys

Rate Limiting with API Keys

APIs use keys to track and limit usage per client:

Rate Limit Headers in Response

HTTP/1.1 200 OK
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 1703001600

{
  "data": [...],
  "message": "You have 847 requests remaining"
}

What this means:

  • Max 1000 requests per hour
  • 847 requests remaining in this hour
  • Resets at Unix timestamp 1703001600

Exceeding Rate Limit

HTTP/1.1 429 Too Many Requests
Retry-After: 3600
X-RateLimit-Remaining: 0

{
  "error": "Rate limit exceeded",
  "message": "You've made too many requests. Please retry after 1 hour."
}

Real-World API Key Examples

🤖 OpenAI API

curl https://api.openai.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk-proj-abc123..." \
  -d '{
    "model": "gpt-4",
    "messages": [{"role": "user", "content": "Hello"}]
  }'

🗺️ Google Maps API

GET https://maps.googleapis.com/maps/api/place/textsearch/json?
query=restaurants%20in%20Sydney&
key=AIzaSyDYwzlRISAM2CmOW3Aj9Eo2VCZQ3o3K8zw

💳 Stripe API

curl https://api.stripe.com/v1/charges \
  -u YOUR_STRIPE_SECRET_KEY: \
  -d amount=2000 \
  -d currency=usd

Basic Auth: colon-separated (key:password where password is empty). Replace YOUR_STRIPE_SECRET_KEY with your actual API key.

💡 Practical Examples: Complete API Interactions

Example 1: Creating a User (POST)

Scenario: Sign up a new user in the system

Request:

POST /api/v1/users
Host: api.example.com
Content-Type: application/json
Authorization: Bearer sk_live_token123
X-Request-ID: req-001-xyz

{
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "password": "SecurePass123!",
  "age": 28
}

Success Response (201 Created):

HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/v1/users/1234
X-RateLimit-Remaining: 499

{
  "id": 1234,
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "age": 28,
  "created_at": "2024-06-18T10:30:00Z",
  "status": "active"
}

Error Response (400 Bad Request):

HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": "ValidationError",
  "message": "Invalid email format",
  "details": {
    "field": "email",
    "value": "alice@invalid",
    "constraint": "must be valid email"
  }
}

Example 2: Fetching User Data (GET)

Scenario: Get a specific user's profile

Request:

GET /api/v1/users/1234
Host: api.example.com
Authorization: Bearer sk_live_token123
If-Modified-Since: Wed, 17 Jun 2024 00:00:00 GMT

Success Response (200 OK):

HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: private, max-age=300
ETag: "abc123def456"

{
  "id": 1234,
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "age": 28,
  "role": "user",
  "created_at": "2024-01-15T08:00:00Z"
}

Not Modified Response (304 Not Modified):

HTTP/1.1 304 Not Modified
ETag: "abc123def456"
Cache-Control: private, max-age=300

User still has the same cached version, no need to re-download

Example 3: Updating User Data (PATCH)

Scenario: Update only the user's email

Request:

PATCH /api/v1/users/1234
Host: api.example.com
Content-Type: application/json
Authorization: Bearer sk_live_token123

{
  "email": "alice.new@example.com"
}

Success Response (200 OK):

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1234,
  "name": "Alice Johnson",
  "email": "alice.new@example.com",
  "age": 28
}

Example 4: Handling Unauthorized Access (401)

Scenario: API key is missing or invalid

Request (missing auth):

GET /api/v1/users/1234
Host: api.example.com

Error Response (401 Unauthorized):

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="API"
Content-Type: application/json

{
  "error": "Unauthorized",
  "message": "Missing or invalid API key",
  "hint": "Include 'Authorization: Bearer YOUR_KEY' header"
}

Example 5: Handling Forbidden Access (403)

Scenario: User is authenticated but lacks permission

Request (valid auth, insufficient permission):

DELETE /api/v1/users/5678
Authorization: Bearer sk_live_token123

(User 1234 trying to delete another user 5678)

Error Response (403 Forbidden):

HTTP/1.1 403 Forbidden
Content-Type: application/json

{
  "error": "Forbidden",
  "message": "You don't have permission to delete this user",
  "required_role": "admin"
}

Example 6: Handling Rate Limiting (429)

Scenario: Made too many requests

Request (50th request in a minute, limit is 30):

GET /api/v1/data
Authorization: Bearer sk_live_token123

Rate Limit Response (429 Too Many Requests):

HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 30
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1703001630
Content-Type: application/json

{
  "error": "RateLimitExceeded",
  "message": "You've exceeded your rate limit",
  "retry_after_seconds": 30
}

Client should wait 30 seconds before retrying

📌 Key Takeaways: Session 2 Summary

1. HTTP Methods

GET (read), POST (create), PUT (replace), PATCH (update), DELETE (remove), HEAD, OPTIONS. Each has specific semantics and idempotency guarantees.

2. Status Codes

2xx (success), 3xx (redirect), 4xx (client error), 5xx (server error). Status codes tell clients what happened with their request.

3. Headers

Metadata in requests and responses. Authorization, Content-Type, Cache-Control, CORS headers, and custom headers enable communication details.

4. API Keys

Credentials for authentication and access control. Keep secret keys secure, use environment variables, and monitor usage.

5. Complete Interactions

Real API calls combine methods, headers, status codes, and JSON data to accomplish tasks. Understanding each piece helps you design robust APIs.
