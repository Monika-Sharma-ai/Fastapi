🎯 Introduction: What is an API?

An API (Application Programming Interface) is a set of rules and protocols that allows different software applications to communicate with each other. Think of it as a waiter in a restaurant who takes your order and brings you the food.

Key Definition: An API is a bridge between different applications that allows them to exchange data and request services from each other in a standardized way.

Why Do We Need APIs?

  • Integration: Allows different systems to work together seamlessly
  • Reusability: Use functionality from other applications without rebuilding it
  • Scalability: Build complex systems by combining smaller, specialized services
  • Standardization: Provides a consistent way for applications to communicate
  • Security: Controls what data is exposed and how it's accessed

Real-World Analogy

📱 Mobile App Weather Example

Your weather app doesn't collect data from weather stations worldwide. Instead, it:

  1. Sends a request to a weather service API (like OpenWeatherMap)
  2. Passes your location as a parameter
  3. Receives formatted weather data back
  4. Displays it beautifully on your screen

🖥️ Client vs Server: The Architecture

What is a Client?

A client is any application or device that requests data or services. Examples:

  • Web browsers (Chrome, Firefox, Safari)
  • Mobile applications (iOS, Android apps)
  • Desktop software
  • Another backend server

What is a Server?

A server is a computer or application that responds to requests with data or services. It:

  • Listens for incoming requests
  • Processes the request
  • Sends back a response
  • Runs continuously to handle multiple requests

How They Communicate

│ CLIENT │───Request────────→│ SERVER │
│ (Browser/ │←───Response───────│ (API/ │

Client-Server Model Benefits

Aspect Client Server
Role Initiates requests Processes and responds
Storage Limited (browser cache, device memory) Large (databases, file systems)
Processing User-side operations Business logic, security, data validation
Examples Web pages, mobile apps, desktop apps Web servers, APIs, databases

📨 Requests vs Responses: The Conversation

Understanding an API Request

A request is what a client sends to a server. It contains:

  • URL/Endpoint: Where to send the request
  • Method: What action to perform (GET, POST, etc.)
  • Headers: Metadata about the request (content type, authentication)
  • Body: Data being sent (optional, usually for POST)
  • Parameters: Query parameters or path variables

📝 Example API Request

GET /api/users/123 HTTP/1.1
Host: example.com
Authorization: Bearer your_token_here
Content-Type: application/json

(Body can be empty for GET requests)

Breaking it down:

  • GET - The HTTP method (retrieve data)
  • /api/users/123 - The endpoint with ID parameter
  • Authorization header - Security token for authentication

Understanding an API Response

A response is what the server sends back. It includes:

  • Status Code: Indicates success/failure (200, 404, 500, etc.)
  • Headers: Metadata about the response
  • Body: The actual data (usually in JSON format)

✅ Example Successful Response

HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 256

{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com",
  "status": "active"
}

Breaking it down:

  • 200 OK - Request was successful
  • Content-Type: application/json - Response format
  • Body contains the requested user data

❌ Example Error Response

HTTP/1.1 404 Not Found
Content-Type: application/json

{
  "error": "User not found",
  "message": "No user with ID 123 exists",
  "status_code": 404
}

Common HTTP Status Codes

Status Code Meaning Example
200 OK - Request successful Data retrieved successfully
201 Created - Resource created New user created
400 Bad Request - Invalid data Missing required field
401 Unauthorized - Authentication required Invalid or missing API key
404 Not Found - Resource doesn't exist User ID doesn't exist
500 Server Error - Server problem Database connection failed

📋 JSON Basics: Data Format

JSON (JavaScript Object Notation) is the standard format for exchanging data in APIs. It's human-readable, lightweight, and universally supported.

JSON Data Types

🔢 Numbers

{
  "age": 25,
  "price": 19.99,
  "temperature": -5
}

📝 Strings

{
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "message": "Hello, World!"
}

✓/✗ Booleans

{
  "is_active": true,
  "is_admin": false,
  "is_verified": true
}

📚 Arrays (Lists)

{
  "hobbies": ["reading", "coding", "gaming"],
  "scores": [95, 87, 92],
  "empty_list": []
}

🏢 Objects (Nested)

{
  "user": {
    "name": "Bob Smith",
    "address": {
      "street": "123 Main St",
      "city": "New York"
    }
  }
}

⚪ Null Value

{
  "phone": null,
  "nickname": null
}

Complete JSON Example: User Profile API Response

💼 Real-world JSON Structure

{
  "id": 1001,
  "name": "Sarah Williams",
  "email": "sarah@example.com",
  "age": 28,
  "is_active": true,
  "roles": ["user", "moderator"],
  "profile": {
    "bio": "Software developer and AI enthusiast",
    "location": "San Francisco, CA",
    "joined_date": "2022-01-15"
  },
  "social_media": {
    "twitter": "@sarah_dev",
    "github": "sarah-williams"
  },
  "phone": null
}

JSON Rules to Remember

  • ✓ Keys must be in double quotes
  • ✓ String values must be in double quotes
  • ✓ Numbers, booleans, and null don't need quotes
  • ✓ Comma separates items but NOT after the last item
  • ✗ Single quotes are NOT valid in JSON
  • ✗ Trailing commas will cause errors
  • ✗ Comments are NOT allowed in JSON

✏️ Exercise: Identify the JSON Errors

Can you spot what's wrong with this JSON? (Hint: there are 3 errors)

{
  'name': 'John', // This is a comment

Answers:

  1. Keys use single quotes instead of double quotes
  2. Comments (// ...) are not allowed in JSON

📌 Key Takeaways: Session 1 Summary

1. What is an API?

An API is a bridge between applications that allows them to exchange data and request services in a standardized way.

2. Client-Server Architecture

Clients make requests, servers process them and send responses. This separation of concerns enables scalable, maintainable systems.

3. Requests & Responses

Every API interaction consists of a request (with method, URL, headers, body) and a response (with status code, headers, body).

4. JSON Format

JSON is the universal language of APIs, supporting strings, numbers, booleans, arrays, objects, and null values.

5. GenAI Applications

Modern AI apps like ChatGPT, search engines, payment processors, and weather apps all rely on APIs to function.

❓ Common Questions & Answers

Q: What's the difference between an API and a website?

A: A website returns HTML/CSS/JavaScript to display in a browser. An API returns structured data (usually JSON) for programs to process. APIs are "headless" (no visual interface).

Q: Can a client be a server?

A: Yes! A backend service can be a client to another API. For example, your FastAPI app might be a client to OpenAI's API.

Q: Why JSON instead of XML or other formats?

A: JSON is lightweight, human-readable, and natively supported in JavaScript and most programming languages. It became the industry standard.

Q: What does "RESTful" mean?

A: REST (Representational State Transfer) is an architectural style for APIs using HTTP methods (GET, POST, PUT, DELETE) on resources. We'll cover this deeply in Session 2.

Q: Is every 200 status code response successful?

A: Yes, 200-299 codes indicate success. 2xx means the server understood and fulfilled the request correctly.

Q: Can I see actual API requests being made?

A: Yes! Open your browser's Developer Tools (F12), go to Network tab, and refresh a website. You'll see all the requests and responses.

Q: Do all APIs use JSON?

A: Not all, but most modern APIs do. Some use XML, Protocol Buffers, or MessagePack. JSON is most common for web APIs.

Q: What's "CORS" and why does it matter?

A: CORS (Cross-Origin Resource Sharing) is a security mechanism. By default, browsers block requests from one domain to another. We'll handle this in FastAPI soon!
